# 基于 STM32 + ESP8266 的 IoT 环境监测系统

> 基于 STM32F103C8T6 + ESP8266 的 IoT 环境监测系统，实现 I2C 传感器驱动、OLED 本地显示、UART AT 指令控制 WiFi 模块、手工裸写 MQTT 协议云端数据上报与云端远程控制。

## 系统框架

```
SHT30
  │ I2C（每 1 秒）
  ▼
STM32F103C8T6
  ├─ OLED 显示温湿度 + 阈值（每 1 秒刷新，按键触发立即刷新）
  ├─ 超阈值报警：PC14 LED 100ms 闪烁
  ├─ 按键设置温湿度阈值（100ms 轮询消抖，长按连续加减）
  │
  │ UART AT 指令（每 10 秒）
  ▼
ESP8266
  │ MQTT（手工裸写报文）
  ▼
云端平台
  ├─ 订阅 stm32/sensor     → 接收温湿度数据
  ├─ 订阅 stm32/threshold  → 接收阈值数据
  └─ 发布 stm32/control    → 下发 LED_ON / LED_OFF 指令
```

## 通信协议

### STM32 ↔ SHT30（I2C）

- 地址：`0x44`（ADDR 接 GND）
- 采集命令：`0x2C06`（单次高精度）
- 返回 6 字节：温度 2 字节 + CRC 1 字节 + 湿度 2 字节 + CRC 1 字节

### STM32 ↔ ESP8266（UART）

- 波特率：115200
- 通过 AT 指令控制：连接 WiFi、建立 TCP 连接、裸写 MQTT 报文发布与订阅
- UART 中断接收 + `+IPD` 状态机实时解析云端下发数据

### MQTT 主题与数据格式

| 方向     | 主题              | 格式示例                                          |
| -------- | ----------------- | ------------------------------------------------- |
| 上报     | `stm32/sensor`    | `{"temp":25.60,"humi":60.20}`                     |
| 上报     | `stm32/threshold` | `{"temp_threshold":25,"humi_threshold":60}`       |
| 云端下发 | `stm32/control`   | `LED_ON` / `LED_OFF`                              |

## 主循环任务调度

采用**非阻塞定时（Non-blocking Timer）**模式，`while(1)` 高速轮询，各任务独立计时互不阻塞。

> 核心思想：把「等待」变成「检查」。`HAL_Delay` 是站在那里等闹钟响；`HAL_GetTick()` 是该干嘛干嘛，路过时瞄一眼表。

| 任务             | 触发方式              | 频率        |
| ---------------- | --------------------- | ----------- |
| 处理云端指令     | `ipd_ready` 标志位    | 立即响应    |
| 按键轮询消抖     | `HAL_GetTick()` 定时  | 每 100ms    |
| 超阈值 LED 闪烁  | `HAL_GetTick()` 定时  | 每 100ms    |
| 读取温湿度       | `HAL_GetTick()` 定时  | 每 1000ms   |
| 刷新 OLED        | `oled_need_refresh` 标志位 | 按需立即    |
| MQTT 上报        | `HAL_GetTick()` 定时  | 每 10000ms  |

## 关键设计说明

### ESP8266 硬复位

PA3 连接 ESP8266 RST 引脚。每次 `ESP8266_Init` 先拉低 RST 50ms 再释放，确保模块从干净状态启动，避免上电时序导致的随机初始化失败。

### UART 中断 +IPD 状态机

UART 接收中断中实现统一 `ipd_state` 状态机（0=空闲, 1-5=匹配"+IPD,"前缀, 6=吸收数据）。`rx_buf`（AT 响应）和 `ipd_buf`（云端数据）双缓冲区完全隔离，二进制 MQTT 报文不影响 AT 命令解析。`ESP8266_Send` 是同步阻塞调用，主循环在发送完成前不会检查 `ipd_ready`，自然实现并发保护。

### SEND OK 等待机制

每次 `ESP8266_Send` 发完数据后必须等待 `SEND OK` 再返回，确保 ESP8266 完全空闲。因此两次连续 `MQTT_Publish` 之间无需额外 `HAL_Delay`。

### 按键消抖

采用 `HAL_GetTick()` 非阻塞轮询，每 100ms 采样一次，兼顾消抖和长按连续响应，无需外部中断。

## 开发流程

- [x] **阶段一：传感器驱动**
  - SHT30 I2C 通信，串口打印温湿度数据
  - CRC 校验验证

- [x] **阶段二：本地显示**
  - OLED 显示温湿度、运行时长、阈值

- [x] **阶段三：WiFi 上云**
  - CH340 调试 ESP8266，AT 指令跑通
  - 手工裸写 MQTT 报文，连接公共 broker
  - 云端面板订阅数据上报成功

- [x] **阶段四：云端控制**
  - 云端面板下发 `LED_ON` / `LED_OFF` 指令
  - ESP8266 订阅 `stm32/control`，`+IPD` 状态机解析
  - STM32 串口接收指令控制 PC13 LED

- [x] **阶段五：本地报警与阈值设置**
  - 超阈值触发 PC14 LED 100ms 闪烁报警
  - 4 个按键本地设置温湿度阈值，支持长按连续调节
  - 阈值实时显示于 OLED，每 10 秒同步上报云端

- [ ] **阶段六：Flash 存储与断网补传**
  - 断网时将数据缓存到 Flash
  - 重连后自动将历史数据补传云端

- [ ] **阶段七：FreeRTOS 重构**
  - 采集任务：定时读取 SHT30
  - 上报任务：处理 MQTT 收发
  - 显示任务：刷新 OLED
  - Queue 在任务间传递温湿度数据
  - Semaphore 同步报警任务

- [ ] **阶段八：低功耗**
  - 无数据变化时进入 STOP 模式
  - RTC 定时唤醒采集上报

- [ ] **阶段九：收尾**
  - 驱动层与应用层代码分离整理
